# ogentic-audit — On-disk format spec, v0.1

**Status:** Draft (depends on [ADR-0001](../adr/0001-on-disk-format.md))
**Tracks:** [OGE-426 (F3)](https://linear.app/ogenticai/issue/OGE-426)
**Last updated:** 2026-05-09

This is the language-agnostic specification of the `ogentic-audit` v0.1 on-disk format. Any conforming implementation — Rust, Python, Go, Node, hand-rolled — must verify the same golden vectors in [`tests/vectors/`](../../tests/vectors/) and produce byte-identical output for the same inputs.

For motivation and rationale, see [ADR-0001](../adr/0001-on-disk-format.md).

## Naming and terminology

| Term | Definition |
|------|------------|
| **Log** | The complete chain of records spanning one or more segments inside a vault |
| **Segment** | A single file on disk (`audit-NNNN.cbor`) containing a contiguous range of records |
| **Record** | A single audit entry with HMAC and chain linkage |
| **Chain** | The HMAC-SHA256 sequence linking record N to record N-1 (and segment N to N-1) |
| **Genesis** | The first record of segment 0; `prev_hash` is 32 zero bytes |
| **Canonical CBOR** | RFC 8949 §4.2 deterministic encoding |
| **Key handle** | An object that exposes `sign(bytes) -> 32-byte HMAC-SHA256` and `key_id() -> 32-byte BLAKE3(public-portion)` |

## File naming

```
<log_dir>/audit-NNNN.cbor
```

- `NNNN` is the zero-padded `segment_index` (4 digits, hex or decimal — implementations MUST support decimal; hex is reserved for v0.2).
- Implementations MUST iterate segments in `segment_index` order.
- Implementations MUST NOT assume segments are contiguous on the filesystem (atomic-rename rotations may briefly create gaps; readers retry once on failure).

## Segment header

The first 80 bytes of every segment are a fixed-layout header:

| Offset | Length | Field | Type | Notes |
|--------|--------|-------|------|-------|
| 0 | 4 | `magic` | bytes | ASCII `"OGAU"` |
| 4 | 2 | `version` | u16 LE | Format version. v0.1 = `0x0001` |
| 6 | 2 | `segment_index` | u16 LE | Zero-indexed; matches filename |
| 8 | 32 | `key_id` | bytes | BLAKE3-256 hash of the public portion of the signing key |
| 40 | 32 | `prev_final` | bytes | Genesis (`segment_index == 0`): 32 zero bytes; else: HMAC of last record in segment N-1 |
| 72 | 4 | `header_crc` | u32 LE | CRC32 (IEEE 802.3 polynomial) of bytes `[0, 72)` |
| 76 | 4 | `reserved` | bytes | Zero-filled, reserved for v0.2 |

Implementations MUST verify `header_crc` on open and reject the segment if it does not match.

## Record framing

Each record is laid out as:

| Length | Field | Notes |
|--------|-------|-------|
| 4 | `len_prefix` | u32 LE — byte length of `payload` |
| `len_prefix` | `payload` | Canonical CBOR encoding of the record map |
| 32 | `hmac` | HMAC-SHA256 of `payload` bytes under the segment's key |
| 4 | `len_trailer` | u32 LE — MUST equal `len_prefix` |

Total record length on disk: `4 + len_prefix + 32 + 4` bytes.

Implementations MUST verify `len_trailer == len_prefix` while reading and treat a mismatch as a torn-tail recovery condition (see Crash Recovery below).

## Record schema

Records are CBOR maps with integer keys. The key set is fixed at v0.1; future versions may add optional keys with integer values ≥ 100.

| Key | Field | CBOR major | Notes |
|-----|-------|-----------|-------|
| 1 | `record_id` | unsigned int (u64) | Monotonic per segment, starts at 0 |
| 2 | `prev_hash` | byte string (32) | Zero on genesis (segment 0, record_id 0); otherwise HMAC of preceding record |
| 3 | `ts_wall` | text string | RFC 3339 UTC, fixed millisecond precision (`2026-05-09T19:48:23.456Z`); MUST end with `Z` |
| 4 | `ts_mono_delta` | unsigned int (u64) | Milliseconds since `session_id` start on a monotonic clock |
| 5 | `session_id` | byte string (16) | UUIDv4; constant for all records in a session |
| 6 | `actor` | text string | Implementation-defined namespacing recommended (`user:david`, `system:shield`) |
| 7 | `event` | text string | Short stable tag in `category.action` form (e.g. `vault.unlocked`) |
| 8 | `payload` | map | Event-specific structured data, CBOR map with text-string keys |
| 9 | `key_id` | byte string (32) | MUST equal segment header `key_id` |
| 10 | `schema_version` | unsigned int (u8) | Major version of the `payload` schema for `event` |

All fields are required. Unknown integer keys MUST be rejected at v0.1 (forward compatibility is via additive optional keys ≥ 100 in later versions, gated by `version` in segment header).

### Canonical encoding rules

Per RFC 8949 §4.2 (Deterministic Encoding):

1. Every integer MUST use the shortest possible encoding (no extra zero-pad bytes).
2. Map keys MUST be sorted by length, then by lexicographic byte order.
3. All length-prefixed items MUST use definite-length encoding (no indefinite items).
4. Strings MUST be UTF-8 with shortest valid encoding.
5. No unused tags. Tag numbers reserved by IANA but irrelevant to the schema MUST NOT appear.
6. Floats MUST NOT appear in v0.1. (No record fields use float; payload maps SHOULD avoid them.)

Implementations MUST validate canonical-form on read and treat violations as a `RecordCorrupt` violation.

## HMAC chain

```
hmac_0  = HMAC-SHA256(key, header_bytes_0..72)         ← genesis (segment 0 only)
hmac_n  = HMAC-SHA256(key, payload_bytes_n)            ← n ≥ 0; sets record.prev_hash for n+1
final_n = hmac_{last record} of segment N
```

For segment N+1:
- Header `prev_final` MUST equal `final_N`.
- Record 0's `prev_hash` field MUST equal `final_N`.

Verifier algorithm (pseudocode):

```
let mut prev = HMAC(key, header_bytes_0..72) if segment.index == 0
              else segment.header.prev_final

for record in segment:
    if record.prev_hash != prev:
        report ChainBreak at record.record_id
        return
    expected_hmac = HMAC(key, record.payload_bytes)
    if record.hmac != expected_hmac:
        report HmacMismatch at record.record_id
        return
    prev = record.hmac

return Verified(final = prev)
```

## Time anchoring

Every record carries `ts_wall`, `ts_mono_delta`, and `session_id`. The verifier reports a `TimestampInconsistency` violation if, within the same `session_id`, the wall-clock delta and monotonic delta diverge by more than 60,000 ms (one minute slack for NTP corrections).

Across `session_id` boundaries (vault was locked then unlocked), only `ts_wall` continuity is checked because the monotonic clock resets. Verifier MUST report `TimestampRegression` if a record's `ts_wall` is earlier than the prior record's `ts_wall` by more than 60,000 ms.

## Segment rollover

When the writer's accumulated segment size would exceed the configured threshold (default 64 MiB) on the next record append:

1. Writer appends a final `segment.finalized` record:
   ```
   actor:    "system:audit"
   event:    "segment.finalized"
   payload:  { "records": <u64 count>, "final_hash": <32-byte HMAC of preceding record> }
   schema_version: 1
   ```
2. Writer fsyncs the segment file.
3. Writer creates `audit-NNNN+1.cbor` with header `prev_final = hmac of segment.finalized record`.
4. Subsequent appends go to the new segment.

Readers MUST iterate segments transparently across rollovers. The chain is one continuous sequence regardless of segment count.

## Crash recovery

Power-loss / SIGKILL between `len_prefix` and `len_trailer` may leave a torn record at the tail of the current segment.

On `Writer::open(path)`:

1. Read segment header; verify `header_crc`.
2. Walk records from offset 80 forward, reading `len_prefix`, validating `len_trailer == len_prefix`, validating HMAC.
3. The last record passing all three checks is the resume point; truncate the file to `(resume_offset + record_size)`.
4. Resume appending from the truncated boundary.

If the segment header itself fails CRC, the segment is unrecoverable and must be quarantined (renamed to `audit-NNNN.cbor.corrupt`); the writer falls back to the prior segment's last record as the chain head.

Truncation MUST be atomic with respect to other readers (use platform-appropriate `fsync` after truncate).

## Violation taxonomy

The verifier ([R3 / OGE-437](https://linear.app/ogenticai/issue/OGE-437)) MUST report exactly one violation kind, the first one encountered:

| Kind | Trigger |
|------|---------|
| `HmacMismatch` | Computed HMAC over `payload_bytes` does not equal `record.hmac` |
| `ChainBreak` | `record.prev_hash` does not equal preceding record's HMAC (or genesis condition) |
| `RecordCorrupt` | `len_trailer != len_prefix`, payload not canonical CBOR, or schema validation fails |
| `TimestampRegression` | `ts_wall` regresses across or within a session by > 60,000 ms |
| `TimestampInconsistency` | Within a session, `\|wall_delta - mono_delta\| > 60,000 ms` |
| `SegmentDiscontinuity` | Segment N+1's `prev_final` does not equal segment N's last record HMAC |
| `KeyIdMismatch` | A record's `key_id` field does not equal its segment header `key_id` |
| `HeaderCorrupt` | Segment header CRC32 fails |
| `UnknownVersion` | Segment header `version` is not 0x0001 |

Violations include enough evidence (record_id, segment_index, expected vs actual bytes) for an auditor to reproduce the failure. Exact JSON shape: see [`docs/spec/violation-report.md`](violation-report.md) (TBD).

## Golden vectors

`tests/vectors/v0.1/` contains golden vectors exercising every code path. Each vector is a directory:

```
tests/vectors/v0.1/<vector-name>/
  ├── README.md          # human description: what does this vector test?
  ├── key.hex            # 32-byte HMAC key, hex-encoded
  ├── inputs.json        # ordered list of records to write (timestamps as ISO strings)
  ├── audit-0000.cbor    # expected segment file bytes
  ├── audit-0001.cbor    # ... if applicable
  └── chain.json         # expected per-record HMAC values, hex-encoded
```

A conforming implementation MUST:
- Given `key.hex` and `inputs.json`, produce byte-identical `audit-NNNN.cbor` files
- Given the same files, produce a `chain.json` that matches byte-for-byte
- Given a tampered version of any segment file (single-byte mutation), report a violation matching the vector's `expected_violation` field

v0.1 ships with at least 6 golden vectors:

1. **happy-path-genesis**: single segment, 5 records, no rollover
2. **rollover-segment**: 2 segments, ~64 MiB each
3. **session-boundary**: 2 records in session A, 3 records in session B (different `session_id`)
4. **tamper-record-payload**: tamper byte in record 2's payload → expect `HmacMismatch`
5. **tamper-prev-hash**: tamper byte in record 2's `prev_hash` → expect `ChainBreak`
6. **clock-tampering**: monotonic and wall-clock disagree by 5 minutes → expect `TimestampInconsistency`

[OGE-441 (Q2)](https://linear.app/ogenticai/issue/OGE-441) tracks the cross-language vector suite.

## Versioning

The `version` field in the segment header is a major-version number. A breaking change (record schema, canonical-form rules, HMAC algorithm) increments `version`. Readers MUST refuse segments with unknown `version` rather than guess.

Backward-compatible additions within a major version use additive optional keys ≥ 100 in the record map. v0.1 reserves keys 1–10; v0.1 readers MUST reject records containing keys 11–99.

## Open questions (v0.1 spike)

These resolve before R1 starts:

1. **Canonical-form parity in `ciborium` and `cbor2`** — verify both produce byte-identical canonical encoding for our schema. (One-day spike per ADR-0001 Action item 4.)
2. **`segment_index` width** — u16 caps segments at 65,536. At 64 MiB / segment, that's 4 TiB of audit log per key — comfortable for v0.1 but worth flagging.
3. **CBOR canonical-form support in target Python lib** — `cbor2`'s `CBOREncoder(canonical=True)` parameter handling under different number sizes.
