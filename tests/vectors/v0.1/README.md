# ogentic-audit golden vectors — v0.1

Language-agnostic conformance vectors for the v0.1 on-disk format. Specified in [`docs/spec/v0.1.md`](../../../docs/spec/v0.1.md); generated by [`tools/gen_vectors.py`](../../../tools/gen_vectors.py).

A conforming implementation, given the same `inputs.json`, MUST produce byte-identical `audit-NNNN.cbor` files (after applying any `post_process` step) and a byte-identical `chain.json`. A conforming verifier, when reading the on-disk segment files, MUST return a verdict equal to the vector's `expected_verdict`.

## Layout

```
v0.1/
  empty/                 segment 0, header only (no records)
  single-record/         segment 0, one record
  1k-records/            segment 0, 1000 records (generated deterministically)
  tampered-byte/         5 records, then byte 50 of record 2's payload XOR'd
  missing-record/        5 records, then record 2 surgically removed
  segment-rollover/      writer with segment_size_bytes=1024 → multiple segments
```

Every vector directory contains:

| File | Purpose |
|------|---------|
| `inputs.json` | The vector's inputs: 32-byte HMAC key (hex), session UUID, ordered record list (or generator config), writer config, optional `post_process` step, expected verifier verdict |
| `audit-NNNN.cbor` | Expected segment file bytes — *post-tampering*, if any. This is what the verifier reads. |
| `chain.json` | Expected key_id, per-segment chain_start and final HMAC, per-record HMAC and prev_hash — all reflecting the *pre-tamper* chain. Plus `expected_verdict` and `post_process` mirror for cross-check. |

## Why two views (`audit-*.cbor` post-tamper; `chain.json` pre-tamper)?

The two views serve two different consumers:

- The **verifier**, given just the on-disk `audit-*.cbor` files, must report `expected_verdict`. For tamper vectors, the on-disk files have been mutated; the verifier detects the mutation.
- The **encoder/byte-conformance test**, given `inputs.json`, must independently re-derive the chain. `chain.json` is what its untampered build should produce internally before the test harness applies `post_process` to compare against the on-disk files.

This lets a single vector exercise both the writer's encoding correctness and the verifier's detection logic.

## Verdict format

`expected_verdict` is one of:

- `"Verified"` — chain is intact end-to-end.
- `"<ViolationKind>@s<segment>r<record_id>"` — verifier reports `<ViolationKind>` at segment `<segment>`, record_id `<record_id>`. Kinds are enumerated in [`docs/spec/v0.1.md`](../../../docs/spec/v0.1.md#violation-taxonomy).

## Regenerating

```sh
# inside a venv with the dev dependencies
pip install blake3 cbor2

python3 tools/gen_vectors.py             # rewrite all vectors
python3 tools/gen_vectors.py --check     # exit non-zero if any output drifted
python3 tools/check_cbor_parity.py       # validate canonical-form parity vs cbor2
```

`--check` is intended for CI: any drift between the committed bytes and what the generator produces today blocks merge. This is how we keep the spec, the generator, and the wire format in lockstep.

## Cross-language conformance

[OGE-441 (Q2)](https://linear.app/ogenticai/issue/OGE-441) wires these vectors into a Rust + Python test harness. Both implementations:

1. Read `inputs.json` and rebuild the segment bytes — must equal the on-disk files (after `post_process`).
2. Run the verifier on the on-disk files — must return `expected_verdict`.

Vectors are append-only within a format version. If a future v0.2 changes the wire format, vectors land under `tests/vectors/v0.2/` and the v0.1 directory remains unchanged so v0.1 readers continue to compile and pass.
